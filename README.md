# guardrails-kit — LLM の不確実性を、確定実行されるコードで置き換えていくレール

**主目的**: LLM の不確実性を排除するために、テスト・pre-commit・CI という**確実性のある
コードの記述と確定実行**によって、信頼できる領域——守られているかを信じる必要がなく、
機械が判定した事実だけがある領域——を**確実に増やしていくレール**を敷くこと。

LLM エージェント（Claude Code / Codex / Cline 等）の出力は確率的で、同じ失敗が繰り返される:
直したはずのバグの再発、テストの省略、規約の風化、「完了しました」という虚偽の自己申告。
プロンプトに規律を書いても、**明文の規則ですら AI は破る**（Martin Fowler の SDD 検証、
EPAM の constitution 事例——`surveys/SURVEY_FAMOUS_KITS.md`）。プロンプトは確率を
上げるだけで、確実にはしない。

対して、テスト・git フック・CI は**書いた通りに毎回実行されるコード**だ。規律を1つ
プロンプト（心得）から門へ移すたびに、「守られていると信じる」対象が1つ減り、
「機械が判定した事実」が1つ増える。このキットはその置き換えを、編集直後→コミット→
push→CI の全工程で、**検査自身の検証**（違反注入・門番のコーパス回帰・red-first 証明・
テストのオラクル契約）まで含めて積み上げるためのレール一式である。

ただし、排除できる不確実性には**原理的な境界**がある——機械が確定判定できるのは
「機械検査可能な違反」まで（要件の誤り・設計の失敗・共通の仕様読み違えは残る——
「何ではないか」と `.guardrails/GUARDRAILS.md` §2e・§9.6 が境界の正本）。このキットの
仕事は、境界を正直に引いた上で、**境界の内側を1規則ずつ広げること**。境界の外を
「守れているはず」と装わないこと自体が、このキットの設計原則である。

## 想定読者

1. **LLM エージェントで長寿命のプロジェクトを回していて、出戻りに焼かれたことがある人**
   ——道具として。全部でなく、部品単位（red-first 証明器・ブートストラップ監査器・
   迂回ガードのコーパス回帰）の持ち帰りも設計上想定しています。
2. **設計判断の記録として読む人**——本リポジトリは「動くツール」であると同時に、
   採用・不採用・保留を根拠つきで記録した**エンジニアリング判断のノート**です。
   何を作ったかより「なぜ作らなかったか」に紙幅を割いています（読みどころは後述）。

## 30秒で分かる、このキットの考え方（設計判断の例を1つ）

「バグ修正にはテストを同梱せよ」という規律は多くのチームにある。しかしそれを機械化
（fix コミットにテスト変更が無ければ拒否）しただけでは足りない——**最初から緑のテスト**を
同梱すれば門は通り、そのテストはバグを何も再現していない。そこでこのキットの CI は、
**同梱テストを親コミット（バグがまだ在る状態）に対して実行し、赤だったことを機械証明**する
（`red-first` ジョブ・`scripts/check_red_first.py`）。CI 上で赤にできない修正のための逃げ道は
`RED-FIRST-EXEMPT: 理由`——ただし**理由なしの免除は不成立**（機械）で、免除の乱用点検は
レビュー規約（人間）に明示的に割り当てる。門で塞げる部分・書式で塞げる部分・人間にしか
判定できない部分を切り分けて、それぞれの持ち場に置く——全機構がこの調子で設計されています。

同型の例をもう一つ: エージェントの「完了しました」は信用しない。セットアップの進捗台帳
`.guardrails/BOOTSTRAP.md` の ✅ は、✅ 化しようとするコミットの瞬間に**監査器が該当 Step の検証を
再実行**し、通らなければ物理的に積めません（`scripts/check_bootstrap.py`——虚偽✅・
順序スキップ・まとめ報告の門）。

## 何をしてくれるか（防壁の層）

| 層 | いつ | 代表的な門 |
|---|---|---|
| 編集直後 | Claude Code / Codex の編集フック | 整形→lint の即時実行（違反はその場で差し戻し） |
| 編集直前 | 同 SessionStart + PreToolUse | **所有権ガード**——セッション開始時点で人間の未コミット変更があったファイルへの AI の Edit/Write をブロック（人間が commit / stash すれば自動解除） |
| 操作直前 | 同 PreToolUse | `--no-verify` / force push / フック付け替え / 破壊的 reset・作業消失コマンドの**技術的ブロック**＋事前照会 `dev.py probe` |
| コミット時 | pre-commit / commit-msg | 秘密検出・索引 `STRUCTURE.md` の鮮度（自動生成と diff）・構造検査（レイヤー違反 / ログ直呼び / flaky パターン / 非推奨 API / UI テストID / `.env` 追跡拒否 / 常時読込文書の肥大 / 孤立ファイル）・**バインディングの刻印一致と死活**（充填値がファイル移動・改名で現実から剥がれ検査が静かに不発になるドリフトの検出）・**fix⇔回帰テスト対**・feat⇔設計根拠対・依存の黙認追加禁止・MCP 許可リスト・コミット規模 |
| プッシュ時 | pre-push | テスト・静的解析（重い検査）＋迂回ブロッカー自身のコーパス回帰再生 |
| CI | GitHub Actions | 全門の再実行（迂回の最終防衛線）＋ **red-first 証明**（上述）＋ **commit-msg 検査の履歴再実行**（PR 範囲の各コミットを「今まさに commit しようとしている状態」に復元して再検査——ローカルフック未導入でも門が掛かる） |
| ターン終了 | Claude Code / Codex Stop フック | 未コミット作業 or 検査赤のままの「終わりました」を差し戻し |
| 導入時 | `check-bootstrap` | 進捗台帳の ✅ を再実行検証（上述） |
| 常時（記録） | 違反ログ | 門が**止めた**事象・無視された soft 警告を JSONL に機械記録（gitignore 済みローカル telemetry）。soft 規則を hard へ昇格する判断の頻度・偽陽性率を逸話でなく計数で行うための土台——記録は機械・解釈と昇格は人間 |

変更の合流は **PR 必須**です。既定ブランチへの直接 push は禁止し、サーバー側の
「PR 必須」と required checks（`checks`・`red-first`・`commit-msg-history`・
`workflow-integrity`・採用した全言語別job）を有効にします。`workflow-integrity` は既定ブランチの
信頼済みコードで PR を検査し、CI workflow 自体の変更を拒否します。ただしGitHubのrequired
checkはworkflowを識別せずjob名だけを見るため、`.github/CODEOWNERS` とサーバー側の
code owner review必須化も併用し、workflowだけでなくPR headで実行するゲート設定・スクリプト・
コーパスの骨抜きと、同名ダミーjobによる偽装を人間境界で止めます。
Step 9はGitHubのCODEOWNERS errors APIも実測し、ownerの不存在・write権限不足を検出します。
一人開発では「人間を2人」にする必要はありません。本人をCODEOWNERにし、PR作成だけを
権限を絞ったmachine user/GitHub Appへ分離します。AIへ本人のowner資格情報は渡しません。
PR は、Web 編集やフック未導入マシンの変更を既定ブランチへ入る**前**に止める境界です。
`push` の CI は合流後の異常検知であり、この予防を代替しません。

**このキットが何をできるかの全一覧は、文書ではなくコマンドで見る**:
`uv run scripts/dev.py gates`——門の台帳（約60機構）を、**そのリポジトリでの実状態**
（常時有効／充填済み／未充填／フック配線済み）つきで表示します。台帳は手書きの機能一覧では
なく機械可読データで、検査器コードとの一致自体が hard 検査（`gates-registry-drift`）で
守られています——機能一覧が実装とずれたらコミットが止まる。発見の経路は3段:
**README（主目的と考え方）→ `dev.py gates`（実状態つき全機能）→ GUARDRAILS 各節（契約の詳細）**。

検査規則の全一覧（規則ID付き）の正本は `.guardrails/GUARDRAILS.md` §3.3——この表は層の索引で、
規則を複製しない。門自身も守られます: 迂回ブロッカーは `tests/guard_corpus.tsv` のコーパス
再生で回帰テストされ、規則を足したら同一コミットでコーパスにも行を足すことが機械検査されます。

静的な門と直交して、開発ループ中は**ランタイム契約**（`.guardrails/GUARDRAILS.md` §12）が土台になります:
全プロジェクト同名の共通動詞
`uv run scripts/dev.py <up/reset/seed/time/test/e2e/fmt/check/probe/db/selftest/doctor>`
（未配線は明示エラー——静かに別コマンドで代替しない。`selftest` = 門の違反注入コーパス
一括、`doctor` = 環境診断、`probe --live` = 実セッションでフックが本当に発火しているかの
sentinel 実測——v2.42）と、実 UI を触る操作レール・コンソール / ネットワーク / ログを読む
観察レールの配線先を、言語ごとのカタログ列が定義します。

## 設計原則（どの機構がどの原則の実装か）

- **門 > 心得**: 機械検査できる規律をプロンプトに書かない。書くのは機械化できない残余だけ。
- **単一の正（G5）**: 同じ情報を2箇所に持たない。規約は AGENTS.md へ二分（複製でなく分割——
  同期スクリプトを作らない理由）、生成物 STRUCTURE.md は「追跡されていること自体が
  鮮度検査の前提」なので gitignore しない。
- **fail-closed と fail-open の使い分け**: 破壊を止める門は fail-closed（インストーラの
  CONFLICT・迂回ブロック）、利便のための門は fail-open（Stop ゲートは環境不足なら1行
  表示して縮退——門が壊れた日に開発を殺さない）。どちらに倒すかを機構ごとに明記。
- **完了 = 実行結果**: 「わざと違反して落ちるのを見た」だけを完了と定義する。全機構の
  DoD は違反注入込みで実測され、`.guardrails/GUARDRAILS.md` の各 Phase 節に記録済み。
- **不採用も記録する**: 検討して入れなかったもの（Serena・GitHub MCP・マルチエージェント
  構成・プロンプト分割ほか）を理由つきで残し、将来の再提案ループを塞ぐ。復活条件は
  トリガーとして登録する。
- **近似は仕様**: 正規表現の検査は完全性を狙わず、偽陽性と検出力のトレードオフを
  明示して境界を文書化する。

## 読みどころ（3分で判断の質を見るなら）

- `surveys/` — **調査の判定記録7本**（MCP・エコシステム／著名キット比較／ゼロレビュー
  系／門アーキテクチャ／実企業の門運用／LLMテスト生成・オラクル系／外部ハーネスツール）。
  「なぜ入れなかったか」を採用ゲート3条で判定した過程の正本。例: Serena を退けた3つの
  理由、自己治癒ランタイムが「門の外の変更経路の極致」である理由、マージスキューという
  **ローカル門では原理的に守れない層**の記録、Meta ACH と同型へ独立収斂した違反注入の裏書き。
- `.guardrails/GUARDRAILS.md` §10 の Phase 節 — 各機構の契約・違反注入の実測・境界の1行明示。
- `scripts/check_bootstrap.py` — 虚偽✅の再実行監査。約250行で読み切れます。
- `README_SETUP.md` の各版セクション — 変更だけでなく**是正の記録**（自分のミスの検出と
  修正——検査番号の欠番、規則IDの記載漏れ等——も監査結果として残しています）。
- `.guardrails/CUSTOMIZE.md` — 導入後にカスタムできる項目の索引。ブートストラップ完了後の入口。

## 何ではないか（正直な注意書き）

- **速くはなりません。** 門は摩擦です。賭けているのは速度ではなく「出戻りの分散の縮小」で、
  それが黒字になるのは長寿命のプロジェクトだけ。週末プロジェクトには過剰です。
- **効果は未実測です。** 各門が違反注入で落ちることは全て実測済みですが、「プロジェクト
  全体のバグが減る」ことはまだ測っていません。キット自身の認識論（完了=実行結果）に
  照らせば、**このキットはまだ自分の基準を満たしていない**——効果は主張しません。
- **機械検査できるものしか守れません。** 要件の誤り・設計の失敗はプロンプト層
  （Superpowers 等——競合ではなく補完）と人間の領分です。
- **敵対的エージェントを拘束するセキュリティ境界ではありません。** 表の「技術的ブロック」は
  現在のセッションの操作経路に対するもので、門の実装ファイル自体への書込みには開いています
  （存在の消失・単体の骨抜きは検出されますが、検査のオラクルごと同一コミットで書き換える
  経路は残りますが、PR 内での CI workflow 自己改変は既定ブランチから動く
  `workflow-integrity`＋必須code owner reviewが止めます）。対象は事故・怠慢・
  機会主義的な近道まで——敵意への耐性はサーバー側設定（ブランチ保護 / rulesets /
  CODEOWNERS——required checks の登録は Step 9 で機械検証）と人間レビューの領分です。
  境界の正本は `.guardrails/GUARDRAILS.md` §2e（脅威モデル）。
- 編集直後・操作直前・ターン終了の3層は **Claude Code と Codex** をサポートする。
  Codex は `.codex/hooks.json` を使い、編集ツール `apply_patch` のパッチから対象ファイルを
  抽出するアダプタを経由する（状態は `.codex/session/` に保持し、gitignore済み）。Cline 等では規約の正本 `AGENTS.md`（ネイティブ読込）＋
  コミット/プッシュ/CI の門（git フックなので全エージェント共通）が同じ規律を担う。
- Codex の PreToolUse はすべての操作経路を捕捉する完全な強制境界ではない。そのため、Codex
  フックは即時フィードバック層として使い、commit / push / CI の門を外さない。

## 使い始める

**前提**: git / [uv](https://docs.astral.sh/uv/) / `uv tool install pre-commit==4.6.0`。
Claude Code / Codex はフック層（編集直後・編集/操作直前・ターン終了）を使う場合のみ
（旧版で必要だった bash・jq は v2.23 のフック Python 化で不要になりました）。Codex は
導入後に `/hooks` でプロジェクトフックをレビューして信頼する。

1. Release の zip を対象リポジトリの**ルートに「置くだけ」**（直下展開・手動コピーは
   禁止——既存の `.gitignore`・CI・pre-commit 設定等と衝突事故になる）。
2. `python3 -m zipfile -e guardrails-kit-*.zip .guardrails-kit-src` →
   `scripts/install_kit.py` を探して実行（GitHub の Download ZIP / Release zip は
   `<repo>-<ブランチ/タグ>/` で1階層ネストされるため固定パスで決め打ちしない——
   `python3 -c "import glob,subprocess,sys; hits=glob.glob('.guardrails-kit-src/**/scripts/install_kit.py',recursive=True); sys.exit('見つからない' if not hits else subprocess.run([sys.executable,hits[0]]).returncode)"`）。
   インストーラが fail-closed の
   衝突判定（INSTALLED / MERGED / KEPT / CONFLICT）で配置し、CONFLICT 0 件の成功時だけ
   zip と展開元を片付けます（冪等）。Claude Code に任せるなら zip を置いて 3 へ。
3. Claude Code に任せるなら新規は `PROMPT_claude_code.md`、既存プロジェクトは
   `PROMPT_claude_code_existing.md`。Codex なら `PROMPT_codex.md` を渡す。以降は `.guardrails/BOOTSTRAP.md` が
   進捗台帳になり、✅ は監査器が再実行検証します（全行 ✅/— で完了）。
   **導入済みリポジトリの新版更新も同じ流儀**——新版 zip をルートに置いて
   `PROMPT_claude_code_update.md` を貼るだけ（インストーラの UPGRADED 判定＋git 履歴の
   安全網で機械配置し、GUARDRAILS §10 の Phase 見出し diff を更新差分の入力にして
   新しい門の DoD まで行う）。
4. 手動で進める場合の正本は `.guardrails/GUARDRAILS.md` §11。導入前に
   `install_kit.py --detect`（採用列の候補提示）と `--diff`（適用プレビュー）、導入後の
   追随確認は `--check`（CI から呼べるドリフト検出——v2.42）。動作確認の最短路は、導入後に
   `uv run scripts/dev.py check` → わざと違反を注入して落ちるのを見ること、そして
   `uv run scripts/dev.py probe --live` で実セッションのフック発火を1回実測すること。

## 読む順番

何ができるかを知るだけなら読まなくてよい——`uv run scripts/dev.py gates` が実状態つきの
全機能一覧（上述）。契約まで読むなら:
`.guardrails/GOALS.md`（目標の正本・14条）→ `.guardrails/GUARDRAILS.md`（契約の正本。困った時は §0 の一次対応表）
→ `README_SETUP.md`（配置手順・全変更履歴・不採用記録）→ `bindings/catalog.md`
（言語固有値の正本: TypeScript/React・Python・Flutter・Rust）→ `surveys/`（判断の出典）。

## このリポジトリ自体の運用

- キットは出荷状態で**自己検査に意図的な指摘5件**を残します（AGENTS.md / CLAUDE.md が
  テンプレのため——移植先の Step 1 で解消。契約は GUARDRAILS §3.3）。導入先では
  うち3件が hard（exit 1）として Step 1 を強制し（残り2件は Step 0 / 3 で解消する
  soft）、キット原本自身のリポジトリだけは原本マーカー `.guardrails-kit-source` により
  3件も SOFT 降格・exit 0 になります（v2.14——原本は雛形のまま配布するのが正しい
  状態のため）。
- 配布は「版 = git タグ」＋ Release。zip はコミットしない（木と zip の二重正本は
  ドリフトの温床——キット自身の G5 が禁じる型）。
- 現行版はリポジトリの **git タグ**が正本（変更履歴と不採用判断の全記録は
  `README_SETUP.md` の各版セクション）。
