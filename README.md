# guardrails-kit — LLM 開発の出戻りを「心得」ではなく「門」で防ぐキット

LLM エージェント（Claude Code / Codex / Cline 等）にコードを書かせると、同じ失敗が繰り返される:
直したはずのバグの再発、テストの省略、規約の風化、「完了しました」という虚偽の自己申告。
プロンプトに規律を書いても、**明文の規則ですら AI は破る**（Martin Fowler の SDD 検証、
EPAM の constitution 事例——`surveys/SURVEY_FAMOUS_KITS.md`）。

このキットの立場は一つ: **機械検査できる規律は、プロンプトではなく git フックと CI の
「門」に置く**。守られるかどうかを信じる必要のない仕組みだけを増やす。

## 想定読者

1. **LLM エージェントで長寿命のプロジェクトを回していて、出戻りに焼かれたことがある人**
   ——道具として。全部でなく、部品単位（red-first 証明器・ブートストラップ監査器・
   迂回ガードのコーパス回帰）の持ち帰りも設計上想定しています。
2. **設計判断の記録として読む人**——本リポジトリは「動くツール」であると同時に、
   採用・不採用・保留を根拠つきで記録した**エンジニアリング判断のノート**です。
   何を作ったかより「なぜ作らなかったか」に紙幅を割いています（読みどころは後述）。

## 30秒で分かる、このキットの考え方（設計判断の例を1つ）

「バグ修正にはテストを同梱せよ」という規律は多くのチームにある。しかしそれを機械化
（fix コミットにテスト変更が無ければ拒否）しただけでは足りない——**最初から緑のテスト**を
同梱すれば門は通り、そのテストはバグを何も再現していない。そこでこのキットの CI は、
**同梱テストを親コミット（バグがまだ在る状態）に対して実行し、赤だったことを機械証明**する
（`red-first` ジョブ・`scripts/check_red_first.py`）。CI 上で赤にできない修正のための逃げ道は
`RED-FIRST-EXEMPT: 理由`——ただし**理由なしの免除は不成立**（機械）で、免除の乱用点検は
レビュー規約（人間）に明示的に割り当てる。門で塞げる部分・書式で塞げる部分・人間にしか
判定できない部分を切り分けて、それぞれの持ち場に置く——全機構がこの調子で設計されています。

同型の例をもう一つ: エージェントの「完了しました」は信用しない。セットアップの進捗台帳
`BOOTSTRAP.md` の ✅ は、✅ 化しようとするコミットの瞬間に**監査器が該当 Step の検証を
再実行**し、通らなければ物理的に積めません（`scripts/check_bootstrap.py`——虚偽✅・
順序スキップ・まとめ報告の門）。

## 何をしてくれるか（防壁の層）

| 層 | いつ | 代表的な門 |
|---|---|---|
| 編集直後 | Claude Code の編集フック | 整形→lint の即時実行（違反はその場で差し戻し） |
| 操作直前 | 同 PreToolUse | `--no-verify` / force push / 破壊的 reset の**技術的ブロック**＋事前照会 `dev.py probe` |
| コミット時 | pre-commit / commit-msg | 秘密検出・索引 `STRUCTURE.md` の鮮度（自動生成と diff）・構造検査（レイヤー違反 / ログ直呼び / flaky パターン）・**fix⇔回帰テスト対**・feat⇔設計根拠対・依存の黙認追加禁止・MCP 許可リスト |
| プッシュ時 | pre-push | テスト・静的解析（重い検査） |
| CI | GitHub Actions | 全門の再実行（迂回の最終防衛線）＋ **red-first 証明**（上述） |
| ターン終了 | Claude Code Stop フック | 未コミット作業 or 検査赤のままの「終わりました」を差し戻し |
| 導入時 | `check-bootstrap` | 進捗台帳の ✅ を再実行検証（上述） |

門自身も守られます: 迂回ブロッカーは `tests/guard_corpus.tsv` のコーパス再生で回帰テスト
され、規則を足したら同一コミットでコーパスにも行を足すことが機械検査されます。

## 設計原則（どの機構がどの原則の実装か）

- **門 > 心得**: 機械検査できる規律をプロンプトに書かない。書くのは機械化できない残余だけ。
- **単一の正（G5）**: 同じ情報を2箇所に持たない。規約は AGENTS.md へ二分（複製でなく分割——
  同期スクリプトを作らない理由）、生成物 STRUCTURE.md は「追跡されていること自体が
  鮮度検査の前提」なので gitignore しない。
- **fail-closed と fail-open の使い分け**: 破壊を止める門は fail-closed（インストーラの
  CONFLICT・迂回ブロック）、利便のための門は fail-open（Stop ゲートは環境不足なら1行
  表示して縮退——門が壊れた日に開発を殺さない）。どちらに倒すかを機構ごとに明記。
- **完了 = 実行結果**: 「わざと違反して落ちるのを見た」だけを完了と定義する。全機構の
  DoD は違反注入込みで実測され、`GUARDRAILS.md` の各 Phase 節に記録済み。
- **不採用も記録する**: 検討して入れなかったもの（Serena・GitHub MCP・マルチエージェント
  構成・プロンプト分割ほか）を理由つきで残し、将来の再提案ループを塞ぐ。復活条件は
  トリガーとして登録する。
- **近似は仕様**: 正規表現の検査は完全性を狙わず、偽陽性と検出力のトレードオフを
  明示して境界を文書化する。

## 読みどころ（3分で判断の質を見るなら）

- `surveys/SURVEY_MCP_ECOSYSTEM.md` — 「話題の MCP をなぜ入れないか」。採用ゲート3条
  （重複排除・常駐トークン予算・契約整合）で Serena / GitHub MCP 等を判定した記録。
- `surveys/SURVEY_ZERO_REVIEW.md` — 「ゼロレビュー本番運用」系（自己治癒ランタイム・
  Dark Factory・Telos 注釈）をなぜ入れないか。「ゼロレビューが買えるのは機械検査可能な
  違反ゼロまで」という立場の固定。
- `surveys/SURVEY_FAMOUS_KITS.md` — Superpowers / Spec Kit / Kiro / BMAD との比較。
  「著名キットの規律はプロンプト層にあり、明文規則でも AI は破る」の外部裏書きと、
  そこから機械化可能な2知見だけを抽出した過程。
- `GUARDRAILS.md` §10 の Phase 節 — 各機構の契約・違反注入の実測・境界の1行明示。
- `scripts/check_bootstrap.py` — 虚偽✅の再実行監査。約250行で読み切れます。
- `README_SETUP.md` の各版セクション — 変更だけでなく**是正の記録**（自分のミスの検出と
  修正——検査番号の欠番、規則IDの記載漏れ等——も監査結果として残しています）。

## 何ではないか（正直な注意書き）

- **速くはなりません。** 門は摩擦です。賭けているのは速度ではなく「出戻りの分散の縮小」で、
  それが黒字になるのは長寿命のプロジェクトだけ。週末プロジェクトには過剰です。
- **効果は未実測です。** 各門が違反注入で落ちることは全て実測済みですが、「プロジェクト
  全体のバグが減る」ことはまだ測っていません。キット自身の認識論（完了=実行結果）に
  照らせば、**このキットはまだ自分の基準を満たしていない**——効果は主張しません。
- **機械検査できるものしか守れません。** 要件の誤り・設計の失敗はプロンプト層
  （Superpowers 等——競合ではなく補完）と人間の領分です。
- 編集直後・操作直前・ターン終了の3層は **Claude Code 限定**。Codex / Cline 等では
  規約の正本 `AGENTS.md`（ネイティブ読込）＋コミット/プッシュ/CI の門（git フックなので
  全エージェント共通）が同じ規律を担います。

## 使い始める

**前提**: git / [uv](https://docs.astral.sh/uv/) / `uv tool install pre-commit` /
bash・jq（ガードコーパス再生に使用）。Claude Code はフック3層を使う場合のみ。

1. Release の zip を対象リポジトリの**ルートに「置くだけ」**（直下展開・手動コピーは
   禁止——既存の `.gitignore`・CI・pre-commit 設定等と衝突事故になる）。
2. `python3 -m zipfile -e guardrails-kit-*.zip .guardrails-kit-src` →
   `python3 .guardrails-kit-src/scripts/install_kit.py`。インストーラが fail-closed の
   衝突判定（INSTALLED / MERGED / KEPT / CONFLICT）で配置し、CONFLICT 0 件の成功時だけ
   zip と展開元を片付けます（冪等）。Claude Code に任せるなら zip を置いて 3 へ。
3. Claude Code に PROMPT を渡す——新規は `PROMPT_claude_code.md`、既存プロジェクトは
   `PROMPT_claude_code_existing.md`（棚卸し→段階有効化）。以降は `BOOTSTRAP.md` が
   進捗台帳になり、✅ は監査器が再実行検証します（全行 ✅/— で完了）。
4. 手動で進める場合の正本は `GUARDRAILS.md` §11。動作確認の最短路は、導入後に
   `uv run scripts/dev.py check` → わざと違反を注入して落ちるのを見ること。

## 読む順番

`GOALS.md`（目標の正本・14条）→ `GUARDRAILS.md`（契約の正本。困った時は §0 の一次対応表）
→ `README_SETUP.md`（配置手順・全変更履歴・不採用記録）→ `bindings/catalog.md`
（言語固有値の正本: TypeScript/React・Python・Flutter・Rust）→ `surveys/`（判断の出典）。

## このリポジトリ自体の運用

- キットは出荷状態で**自己検査が意図的に 5 件赤**（AGENTS.md / CLAUDE.md がテンプレの
  ため——移植先の Step 1 で解消。契約は GUARDRAILS §3.3）。そのためこのリポジトリで
  Actions を有効にすると常に赤になります。無効化を推奨。
- 配布は「版 = git タグ」＋ Release。zip はコミットしない（木と zip の二重正本は
  ドリフトの温床——キット自身の G5 が禁じる型）。
- 現行版はリポジトリの **git タグ**が正本（変更履歴と不採用判断の全記録は
  `README_SETUP.md` の各版セクション）。
