# SURVEY_COMPANY_GATES.md — 調査⑤: 実企業の門運用（2026-07-07・キット v2.18 時点）

> 位置づけ: 「門を採用している企業」の独自調査。調査④（Meta RADAR / StrongDM 等の
> アーキテクチャ）と重複する対象は再判定しない（再提案ループ防止——判定は調査④参照）。
> 本調査の新規発見は**合流の門（マージキュー）**の一系統のみ。結論: 機構としての即時
> 採用ゼロ・保留1件・裏書き複数。

## 1. 企業の門運用（一次事実）

| 企業/プロジェクト | 門 | 事実 |
|---|---|---|
| **Rust（Graydon Hoare）** | bors / Homu（2013〜） | 「Not Rocket Science Rule＝**常に全テストを通るリポジトリを自動維持**」。PR を最新 main と合流した状態でテストし、緑なら main を進める——**マージスキュー**（個別に緑の2 PR が合流で壊れる意味的衝突。例: 片方が関数名を変え、片方が旧名を呼ぶ）を排除 |
| **Uber** | SubmitQueue（投機的合流検証） | 導入前は最悪日で**コミットの10%が revert**・iOS mainline の緑率52%。導入で CI 待ち74%減。2026 に OSS 公開 |
| **Shopify / Strava / Airbnb** | Shipit / Butler / 内製 | 大規模化した組織が同じ結論に収斂（Shopify は日次 ~400 コミット） |
| **GitLab / GitHub** | Merge Trains（2019）/ Merge Queue（2023 GA） | 内製の門が**プラットフォーム標準機能化**——required checks を前提に設定だけで有効化できる |
| Meta / StrongDM / Google TAP | RADAR / Clean Room / 大規模CI | 調査④で判定済み（裏書き・保留）。TAP は上記マージキュー系の着想元として言及される |

## 2. キットとの突合（正直な境界の明示）

- キットの門は「**門を通らないコミットは積めない**」をローカル＋CI で保証する——これは
  NRSR と同じ思想の**コミット粒度**版。しかし NRSR が守る対象は**合流順序**の問題であり、
  **PR 単位の CI では原理的に守れない**（PR の CI が緑でも、base が進めば古い合流状態の
  検証にすぎない）。キットの既設の部分防御 = CI の `push: main` トリガー（合流後に全門を
  trunk 上で再実行——業界通例の後段網と同型。壊れたら**即検知**はするが**予防**はしない）。
- つまりこの層の空白は設計上の見落としではなく**層の違い**: 合流の門はリポジトリ内の
  ファイルでは実装できず、ホスティング側の直列化機構（Merge Queue）が正本になる。

## 3. 判定

- **保留（GUARDRAILS §10 登録）——合流の門（GitHub Merge Queue）**: トリガー =
  **並行 PR の常態化**（複数エージェントの並走・共同開発化）**または合流起因の main 赤を
  1回でも実測**した時。発火時の実装は設定のみ（Step 9 ④で登録済みの required checks が
  そのまま前提になる——キットの全緑シグナルを合流直列化の判定に流用できる）。
  単独開発・低並行では待ち時間コスト > 価値のため発火まで有効化しない。
- **裏書き**: `push: main` 再実行（既設）・「revert 文化より予防の門」（Uber の 10% revert
  という数字は、事後修正文化の実測コストとして記録）。
- **不採用**: マージキューの自作（bors 自前運用等）——プラットフォーム標準機能が存在する
  今、自作は重複排除ゲート違反。ML による投機スケジューリング（Uber）は規模の産物で対象外。
