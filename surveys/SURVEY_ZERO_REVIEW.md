## 0. 結論サマリ
「ゼロレビュー本番運用」に対するキットの立場を1行で固定する:
**ゼロレビューが買えるのは「機械検査可能な違反ゼロ」まで**——RLS の実質・数理の正しさ・
ビジネスロジックの意味は門の射程外に残る（README「何ではないか」と同じ整理）。
レポートの機構群の大半は、①門の外の変更経路を作る（自己治癒）②統治層の外
（オーケストレーション）③形式強制の偽陽性、のいずれかで不採用。採用は**2件＋保留1件**。

## 2. 保留（トリガー登録——GUARDRAILS §10）

- **依存・脆弱性監査の CI ジョブ**（osv-scanner / cargo audit / npm audit 等）: レポートの
  「強制プレパブリッシュスキャン」（Lovable）の門版。トリガー = 対象リポジトリが
  **本番運用・顧客データ段階**に入った時。設計上の緊張を先に記録: アドバイザリ DB は
  日々更新され**同一コミットの CI 結果が時間で変わる**（G1 決定性と衝突）——ゆえに
  非ブロッキング（警告ジョブ）で開始し、運用実測後にブロッキング昇格を判定する。

## 3. 不採用（判断ごと記録——再提案ループ防止）

| 候補 | 不採用の理由 |
|---|---|
| **自己治癒ランタイム**（healing_agent デコレータ: 実行時例外→LLM がソースを書換→ホットリロード） | **門の外の変更経路の極致**（G7）。テスト・コミット・レビュー・履歴の全てを迂回してコードが変わる＝キットの全機構の否定形。再現性（G1）も破壊（同じ入力で同じコードが走る保証が消える）。掲載コード自体に欠陥がある点も象徴的。エラー→修正はキットでは「fix ブランチ＋red-first 証明」という**門の内側の経路**が既にある |
| **Dark Factory**（Planner/Generator/Validator/Orchestrator の4エージェント自動マージ） | 統治層の外＝対象外（調査②の役割エージェント・worktree 並列と同根）。なお **Validator の検証関数 V(Code) はキットの CI そのもの**——「全 required checks 緑で auto-merge」は GitHub 設定1つの人間判断であり、キットは既にその前提（機械検証の全緑）を供給している。機構として足す物が無い |
| **Telos 型の関数単位仕様注釈**（@telos 必須・欠落でコミット拒否・orphan 検出） | 形式強制は偽陽性>価値（調査②の EARS と同じ判定）。同じ目的の粗粒度版をキットは既に持つ: 意図の対（feat⇔plan 検査5・fix⇔テスト検査2・red-first）＋ dead-code は列の静的解析（clippy / analyzer）の担当 |
| **SOUL.md / MEMORY.md / HEARTBEAT.md**（OpenClaw 型ワークスペース） | 中央メモ禁止（§13）・Memory Bank 不採用と同根の再確認。対照が1つ: OpenClaw の .guardrails/BOOTSTRAP.md は**初回後に自己削除**するが、キットの .guardrails/BOOTSTRAP.md は**監査証跡として永続**——儀式ではなく台帳、が設計差 |
| **Vibe Testing**（自然言語→AI ブラウザが自律探索テスト） | **非決定な検証は門になれない**（G1——flaky を落とす側の検査と正面衝突）。AI に Playwright spec を「生成」させ決定的成果物として回す形は既に採用済みの運用（§12.4 の操作レール＋E2E 資産）。探索的 QA としての人間側利用は自由＝キット正本に入れない |
| **トークンバジェット・判断の決定テーブル化**（Unmeshed） | 実行時オーケストレーション＝対象外。ただし「繰り返される LLM 判断を確定的ルールへ縮退」は**キットの中核思想の外部裏書き**（弱いモデル＋強い検証、機械化できる判断は門へ）として記録 |
| SPEC.md 契約・worktree 並列・分散テスト実行 | 調査②の判定を再確認（プロセスファイル/個人ワークフロー層/インフラ層） |
