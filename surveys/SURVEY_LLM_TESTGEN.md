## 0. 結論サマリ

外部の収斂点を1行で固定する: **「テストが緑」を信用しない仕組みが2系統ある——
①テストの検出力を測る（mutation testing: わざと壊したコードを赤にできるか）
②オラクルの独立性を上げる（PBT: 実例の期待値でなく性質を主張する）**。
キットとの対応: ①の門側（違反注入 DoD・guard コーパス §2）と単変異の特殊形
（red-first §5——変異体が「本物の過去のバグ」1個）は実施済み、①の製品側は保留
（§10——トリガー待ち）、**②が v2.41 までのキットの空白**であり今回の採用対象。

## 1. 入力の要点（3系統）

| 系統 | 中身 | キットとの関係 |
|---|---|---|
| **TestGen-LLM**（Meta・FSE 2024） | 既存テストを LLM が改善提案し、**機械フィルタ（ビルド通過・実行安定・カバレッジ純増）を全通過した候補だけ**を人間レビューへ——本番で高受け入れ率 | 「弱い生成＋強い機械検証」の産業実証。キットの中核思想（門>心得）の外部裏書き——機構としては足す物が無い（生成は統治層の外） |
| **ACH**（Meta・2025-02） | mutation-guided: privacy 欠陥を模した **mutant を先に生成**し、それを検出できるテストを LLM に書かせる——「このテストは何を捕まえるか」が構成的に保証される | 「テストをテストする」の実装形。門への同型物（違反注入・guard コーパス）は実施済み。製品側は §10 保留「製品テストへの変異テスト」——本調査で**領域特化形（制約1本壊し）を追記**（採用2） |
| **Property-Generated Solver**（arXiv 2025-06） | LLM 生成テストが実装と欠陥を共有する **self-deception サイクル**を命名し、実例オラクルの代わりに PBT（性質・不変量）で断ち切る。例が制約充足（グラフ彩色——「隣接ノードは異色」は正解表なしで検証可能） | キットの空白だった「オラクルの種類」への直接回答。§9.6 として契約化（採用1）。制約充足領域はこの手法が最も効く土地——本キットの移植元（ソルバー系）と同族 |

## 2. 採用（v2.41 実装）

| 知見 | キットでの形 |
|---|---|
| **オラクルの階層**: 実例オラクル（期待値ハードコード）は self-deception の温床。優先順は 差分検証（独立オラクル）> 性質形（不変量・メタモルフィック関係）> 実例（境界ケース固定に限る） | **§9.6 新設**（契約＋限界の明文化）。機械検査は soft **`missing-property-test`**: 確率的コンポーネント有（`SOLVER_DIRECT_CALL_PATTERNS` 充填）なのに性質形テストの痕跡（`PROPERTY_TEST_MARKERS`）がテストのどこにも無い——存在検査のみ・質は測れない（NO-LOG と同じ境界 — G9）。soft 開始は v2.34 非対称閾値どおり |
| **独立オラクル＝第二参照点は贅沢品**（大半の業務コードには無い。制約充足・参照実装・仕様検証器を持つ領域だけの特権） | **§11 Step 0 表B に「独立オラクルの有無」**を質問として追加——有なら差分検証（§9.6）を配線、無ければ性質形が上限と記録する。質問の前払い（G12） |
| **差分検証の限界**: 共通の定式化ミス（仕様の読み違えが両実装へ同じ形で入る）は原理的に検出不能 | §9.6 に**買うもの/買わないもの**を明文化（§2e 脅威モデルと同じ型）——「LLM の検証関数だけが間違うケース」の検出までが機械の持ち場。仕様との突き合わせはレビュー・ユーザーフィードバックの持ち場 |
| **打ち切りの選択バイアス**: タイムリミット到達を黙って捨てると「難しい問題ほど欠測」する | **§9.1 に UNKNOWN 三値契約**を追記——充足/違反/判定不能で数え、判定不能は表示する（G9 沈黙の禁止のテスト版） |
| **変異体の領域特化形**: 制約充足では「制約を1本壊した定式化」が mutant の自然な単位（ACH の mutation-guided の領域版） | §10 保留「製品テストへの変異テスト」へ**追記**（トリガーは従来のまま——red-first 安定＋CI 予算） |

## 3. 保留・不採用（判断ごと記録——再提案ループ防止）

| 候補 | 判定 |
|---|---|
| **LLM によるテスト自動生成**（TestGen-LLM / ACH 本体の導入） | **不採用**。生成は統治層の外（調査②の役割エージェントと同根）。キットが規定するのは生成物が通るべき門であり、誰が書くかではない。Meta の機械フィルタ相当（ビルド・実行・カバレッジ）はキットの既存 CI が同役 |
| **汎用 PBT の全リポジトリ強制** | **不採用**。強い不変量が書けるのは構造化された領域（制約充足・パーサ・数値変換等）だけ——一般の CRUD/UI に強制すると空虚な性質（`assert result is not None`）で骨抜きになる（missing-log-coverage で全関数強制を退けたのと同じ判定）。だから発火条件を「確率的コンポーネント有」に絞った |
| **UI 第二層への性質検査** | **不採用**（正解が定義されない層——「正しい見た目」の不変量は書けない）。代わりに**変化検出**（visual regression / smoke）を §10 保留へ新規登録（トリガー = 見た目起因の出戻り実測） |

## 4. 裏書きとして記録（機構は足さない）

- **red-first（§5）は mutation testing の単変異・天然物版**——変異体が「本物の過去の
  バグ」であるため人工 mutant より的中率が高い代わりに、fix を経ていないテストの検出力は
  測れない。この非対称が製品側変異テストの保留理由を正当化する（先に red-first、余裕が
  出たら mutant）。
- **違反注入 DoD・guard コーパス（§2）は門に対する mutation testing そのもの**——
  「わざと違反して落ちるのを見届けて初めて完成」（§0）は ACH と同じ原理の門版。
  独立に同型へ収斂したことを外部裏書きとして記録する。
- **探索→昇格ループの型**（運用時観測から検査を育てる回路）: 「広く観測→怪しい候補を
  統計で絞る→局所検証で白黒→check 関数へ昇格→繰り返す」は、キットが既に持つ複利の型
  （違反ログ §3.6 をセンサーに soft→hard 昇格・「心得を検査に変換」）のランタイム版。
  キットに新機構は足さない（領域固有の集計関数は採用先リポジトリの資産——契約と列で
  持てる部分だけをキットが持つ）。採用先で最初の実例が育ったら、骨格（fired-mask 集計・
  候補抽出・局所再検証）を領域列としてカタログへ還元する——列を起こす判断は §12.7。
